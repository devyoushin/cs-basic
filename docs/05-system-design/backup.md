# 백업 전략: 파일 백업 vs 스냅샷 백업

## 파일 백업 (File-level Backup)

파일 시스템을 순회하며 개별 파일을 하나씩 읽어서 복사하는 방식.

```
┌─────────────────────────────┐
│  파일 시스템                  │
│  /var/lib/mysql/             │
│    ├── ibdata1    (읽기)──►  │──► 백업 스토리지
│    ├── ib_logfile0 (읽기)──► │──► 백업 스토리지
│    ├── table1.ibd (읽기)──►  │──► 백업 스토리지
│    └── ...                   │
└─────────────────────────────┘
        ↕ 수천~수만 개 파일 순회
        ↕ 백업 중에도 데이터 변경됨 (불일치 위험)
```

### 문제점

| 문제 | 설명 |
|------|------|
| **느린 속도** | 파일 수만큼 메타데이터 조회(stat) + 파일 열기(open) + 읽기(read) 반복 |
| **일관성 없음** | 백업 도중 다른 프로세스가 파일을 수정하면 백업 데이터가 불일치 |
| **서비스 중단 필요** | 일관성 보장을 위해 DB/애플리케이션을 멈춰야 함 |
| **증분 백업 복잡** | 변경된 파일을 추적하려면 mtime/체크섬 비교가 필요 |

## 스냅샷 백업 (Snapshot Backup)

특정 시점의 볼륨 상태를 **순간적으로 고정**하는 방식. 실제 데이터를 복사하지 않고, 블록 레벨의 포인터만 기록한다.

### 핵심 원리: CoW (Copy-on-Write)

```
스냅샷 생성 직후 (즉각, O(1))
┌──────────────┐     ┌──────────────┐
│  원본 볼륨     │     │  스냅샷        │
│  Block A ────┼─────┼──► Block A   │  (동일 블록을 가리킴)
│  Block B ────┼─────┼──► Block B   │
│  Block C ────┼─────┼──► Block C   │
└──────────────┘     └──────────────┘

원본 데이터 변경 발생 시 (CoW)
┌──────────────┐     ┌──────────────┐
│  원본 볼륨     │     │  스냅샷        │
│  Block A' ◄──┼──   │──► Block A   │  ← 스냅샷은 이전 블록 유지
│  Block B ────┼─────┼──► Block B   │
│  Block C ────┼─────┼──► Block C   │
└──────────────┘     └──────────────┘
      ↑ 새 블록에 기록         ↑ 변경 전 블록 참조 유지
```

1. 스냅샷 생성 = 메타데이터(포인터 테이블)만 복사 → **수 초 이내 완료**
2. 원본 블록이 변경될 때만 이전 블록을 별도 저장 (CoW)
3. 스냅샷은 변경 전 블록을 가리키므로 **그 시점의 일관된 상태** 보장

---

## 왜 스냅샷이 빠른가

### 1. 생성 시간: O(1) vs O(N)

```
파일 백업:   파일 수(N)에 비례 → 수십 GB = 수십 분~수 시간
스냅샷 백업: 메타데이터만 복사  → 수 TB도 수 초 이내
```

### 2. I/O 부하 차이

```
파일 백업 중 I/O 패턴:
  Read(file1) → Read(file2) → ... → Read(fileN)  ← 전체 데이터 읽기

스냅샷 생성 I/O:
  Write(메타데이터 블록)  ← 수 KB만 씀
  이후 변경된 블록만 CoW로 추가 저장
```

### 3. 일관성 (Consistency)

| 방식 | 일관성 | 이유 |
|------|--------|------|
| 파일 백업 | X (기본) | 백업 시작~끝 사이에 파일 변경 가능 |
| 파일 백업 + 락 | O | 서비스 중단 필요 |
| 스냅샷 | O | 특정 시점을 원자적(atomic)으로 고정 |

스냅샷은 **서비스를 중단하지 않고도 일관된 시점의 데이터**를 보장한다.

### 4. RPO (Recovery Point Objective) 단축

```
파일 백업: 시간이 오래 걸려 백업 빈도를 낮춤 → RPO 길어짐 (예: 1일)
스냅샷:   몇 초 만에 생성 → 자주 찍을 수 있음 → RPO 단축 (예: 1시간)
```

## 증분 스냅샷 (Incremental Snapshot)

첫 스냅샷 이후에는 **변경된 블록만** 저장한다.

```
T1 스냅샷: [A][B][C][D][E]          ← 전체 블록
T2 스냅샷: [A'][D']                  ← 변경된 블록만
T3 스냅샷: [B'][E']                  ← 변경된 블록만

복원 T3: T1 기반 + T2 변경 + T3 변경 적용
```

- 스토리지 사용량 최소화
- AWS EBS 스냅샷, GCP Persistent Disk 스냅샷 모두 이 방식

---

## 스냅샷 기술별 구현

| 기술 | 방식 | 사용처 |
|------|------|--------|
| **LVM Snapshot** | CoW, 블록 레벨 | Linux 온프레미스 |
| **ZFS Snapshot** | CoW, 즉각 생성 | FreeBSD, TrueNAS |
| **Btrfs Snapshot** | CoW, 서브볼륨 단위 | Linux |
| **AWS EBS Snapshot** | 증분, S3 저장 | AWS 클라우드 |
| **VMware Snapshot** | VMDK 델타 파일 | 가상화 환경 |
| **DB Snapshot (RDS)** | 스토리지 레벨 CoW | 관리형 DB |

---

## 스냅샷의 한계

| 한계 | 설명 | 대응 |
|------|------|------|
| **Application-level 일관성** | 파일 시스템은 일관되지만 DB 내부 상태는 별도 보장 필요 | DB freeze(fsync) 후 스냅샷, 또는 DB 자체 스냅샷 기능 사용 |
| **장기 보관 비용** | 스냅샷이 쌓이면 델타 블록 누적 → 스토리지 증가 | 보관 정책(retention policy)으로 오래된 스냅샷 삭제 |
| **크로스-리전 복제** | 스냅샷을 다른 리전으로 복사 시 초기 비용 발생 | 증분 복제 활용, 중요 데이터만 크로스-리전 |
| **복구 시간(RTO)** | 대용량 스냅샷 복원은 수십 분 소요 가능 | EBS fast snapshot restore, pre-warming |

---

## 실무 포인트

- **RDS/Aurora**: 스냅샷은 스토리지 레벨 → 수 TB도 수 분 이내 생성, 복원은 별도 인스턴스 기동 필요
- **EBS**: 스냅샷 생성 중에도 볼륨 사용 가능, 복원 후 첫 접근 시 S3에서 lazy loading (pre-warming 필요)
- **DB 백업 시**: `FLUSH TABLES WITH READ LOCK` (MySQL) 또는 `pg_start_backup()` (PostgreSQL)으로 DB를 일시 일관 상태로 만든 뒤 스냅샷 → application-level 일관성 확보
- **3-2-1 규칙**: 3개 복사본, 2개 다른 미디어, 1개 오프사이트(다른 리전/사이트)
- **스냅샷 ≠ 백업**: 같은 스토리지 시스템 장애 시 스냅샷도 함께 유실 가능 → 다른 스토리지로 export 필요
